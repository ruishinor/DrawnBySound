# Drawn by Sound adversarial audit and stabilization plan

Date: 2026-07-11
Scope: uploaded repository state at commit `7feff2e`, excluding `node_modules`.

## Operating assumptions and trade-offs

- The production target is the static Vercel deployment at the site root.
- Mobile means current Chromium/WebKit-class browsers over HTTPS; system-audio capture is not a mobile acceptance requirement.
- Raw audio must remain local. No cloud decoder, analytics, telemetry, or account service is introduced.
- The SharedArrayBuffer architecture remains. A postMessage fallback would be a separate architectural project, not a stabilization patch.
- Browser-native media playback is an acceptable local fallback when `decodeAudioData()` rejects a file.
- The minimum defensible change is preferred over a broad framework or UI rewrite.

## Adversarial retrospective

### Project Lead

The project declared the web MVP complete while its production-only audio path was not covered by a production-bundle test. Development E2E tests ran against Vite's development transform, so they could not detect that the AudioWorklet was emitted as an invalid TypeScript data URL in `dist`. The release gate verified compilation, not executable production behavior.

Primary project-control weakness: acceptance criteria were source-oriented instead of deployment-oriented. The repository had evidence that demo rendering worked, but not that production microphone and file inputs reached the worklet.

### Design Thinking Coach

The interface exposed three different failures as two generic messages: microphone unavailable and file decode failed. Both could actually mean the shared audio engine failed before microphone permission or file decoding was attempted. This prevented users from forming a correct mental model and prevented maintainers from collecting actionable failure evidence.

The product also assumed that one decoding API represented all browser playback capability. That is unnecessary: a local browser media element can sometimes play a file that `decodeAudioData()` rejects.

### UX Designer

On mobile, disabled controls relied on `title` text, which is not a reliable touch interaction. Status text was low-opacity and visually detached from the control that failed. The native file control could accept a file and then appear to do nothing because engine failures were mislabeled as decode failures.

The minimum UX correction is not a redesign. It is visible unavailable labels, an accessible live status region, actionable error copy, and a transport state that remains usable when autoplay is blocked.

### Senior Analyst

The common-cause hypothesis was stronger than independent microphone and MP3 failures because both paths call `ensureGraph()` before their source-specific operation. Inspection of the production bundle confirmed the common cause: `preprocessor.worklet.ts` was embedded as `data:video/mp2t` containing raw TypeScript. `AudioWorklet.addModule()` therefore failed before `getUserMedia()` or `decodeAudioData()` completed.

The previous COOP/COEP patch was necessary but insufficient. It enabled `SharedArrayBuffer`; it did not make the production worklet executable.

## Findings by priority

### P0 — production AudioWorklet bundle was invalid

Confirmed in the uploaded `dist/assets/index-*.js`: the worklet URL was a `data:video/mp2t;base64,...` payload containing TypeScript source. Development mode transformed the module, so existing E2E tests passed while production input failed.

Implemented correction: import the worklet through Vite's `?worker&url` pipeline so production emits a separate executable JavaScript chunk. A post-build verifier now fails if the worklet chunk is absent, does not register the processor, or the invalid MIME data URL reappears.

### P0 — shared engine failures were misclassified

Microphone and file handlers wrapped engine creation and source-specific work in one catch block. A worklet failure became “permission denied” or “couldn't decode.” This made field diagnosis unreliable.

Implemented correction: engine startup, microphone capture, decoding, fallback playback, and autoplay failures now have separate handling and actionable messages. Failed starts stop analysis and return to a safe demo state.

### P1 — AudioContext resume occurred after asynchronous initialization

The graph previously loaded the worklet before explicitly resuming the AudioContext. On stricter
mobile autoplay implementations, that can lose the initiating user gesture before `resume()` is
invoked.

Implemented correction: graph creation invokes `resume()` before its first asynchronous module
load, while still inside the microphone/file interaction path. Microphone startup also exposes
visible engine and permission-request states instead of appearing idle.

### P1 — mobile MP3 resilience was too narrow

The file path depended solely on `decodeAudioData()`. Browser codec support and behavior vary by encoding/container details. A decode rejection had no local fallback.

Implemented correction: if decoded-buffer playback fails, Drawn by Sound tries a same-origin blob URL through an `HTMLAudioElement` connected to the existing Web Audio graph. The fallback remains local, keeps transport controls, and has explicit metadata timeout and cleanup.

### P1 — production input behavior lacked a release gate

The existing Playwright suite used the development server and development-only hooks. It could not prove the built worklet or native file control worked.

Implemented correction: a production-preview Playwright configuration covers WAV, MP3, and a
fake-device microphone at a mobile viewport. CI runs this after the normal development E2E suite.
A static bundle verification runs on every build even when a browser is unavailable.

### P1 — mobile failure communication was weak

Unavailable buttons were greyed out without visible explanatory text, and the status region was low contrast and not announced to assistive technology.

Implemented correction: unsupported microphone/system controls include “unavailable” in their visible label. Status is an `aria-live` region with mobile-specific contrast, safe-area spacing, and responsive transport/settings layout.

### P1 — source failure cleanup was incomplete

Microphone/system capture errors could leave the feature worker running. The previous source object could also remain active briefly after a stop.

Implemented correction: all failure branches call the shared stop path; stopping now immediately selects the silence source. AudioGraph creation closes partially created resources when initialization fails.

### P2 — classifier state did not match the documented source-reset invariant

The feature worker recreated the spectral analyzer on reinitialization but reused the classifier EMA and hysteresis state.

Implemented correction: the classifier and counters reset on each worker initialization.

### Deferred, not hidden

- `main.ts` remains a large composition root. Refactoring it now would increase regression surface without fixing the current bottleneck.
- There is no non-SharedArrayBuffer transport. Browsers that cannot become cross-origin isolated remain demo-only.
- Rapid overlapping source selections are not serialized with cancellation tokens. The UI is small enough that this is lower priority, but it should be addressed before multi-file or playlist features.
- AudioWorklet processor crashes are not surfaced through a dedicated health channel.
- Large compressed files are still attempted through `arrayBuffer()`/`decodeAudioData()` before
  the streaming media-element fallback. A size- or format-aware primary path should be considered
  after representative mobile memory testing, rather than selecting an arbitrary threshold now.
- Microphone tracks ending unexpectedly are not surfaced separately from an intentional Stop.
- The manifest does not imply offline caching: no service worker is registered. Adding one during
  audio-path stabilization would increase stale-deployment risk, so offline installation is a
  separate release decision. Local-first here means local processing, not guaranteed offline use.
- No Content Security Policy is deployed. A CSP should be designed and tested with the inline CSS,
  blob-backed media fallback, worker, and AudioWorklet before enforcement.
- Development dependencies have five audit findings (three moderate, one high, one critical) in
  the Vite/Vitest toolchain. Production has zero runtime dependencies and
  `npm audit --omit=dev` is clean. npm only offers a forced breaking upgrade to address them, so
  the toolchain update should be isolated and validated separately. Until then, do not expose the
  Vite/Vitest development servers to untrusted networks.
- Real-device Android and iOS validation remains mandatory. Automated Chromium preview tests are necessary, not sufficient.

## Modular roadmap and verification gates

### Phase 0 — restore production audio viability

1. Bundle the AudioWorklet through `?worker&url` → verify: `dist/assets/preprocessor.worklet-*.js` exists.
2. Reject invalid production embedding → verify: `npm run build` fails if any built JavaScript contains `data:video/mp2t`.
3. Preserve cross-origin isolation → verify: deployed headers include COOP `same-origin` and COEP `require-corp`.
4. Exercise the native production inputs → verify: production-preview E2E opens both
   `mono-sine.wav` and `mono-sine.mp3`, advances playback without foreign requests, and starts a
   fake-device microphone stream without page errors.

### Phase 1 — make failures diagnosable and recoverable

1. Resume the AudioContext before asynchronous worklet loading → verify: mobile input startup
   calls `ctx.resume()` before `audioWorklet.addModule()`.
2. Separate engine, permission, device, decode, codec, and autoplay errors → verify: unit tests
   cover standard DOMException categories.
3. Clean partial engine initialization → verify: rejected worklet/worker setup closes the
   AudioContext and disconnects created nodes.
4. Stop analysis on failed source start → verify: every microphone, system, and file failure
   branch calls `stopAll()`.
5. Permit retry of the same file → verify: the file input value is cleared after selection.

### Phase 2 — harden mobile file playback

1. Keep AudioBuffer playback as the primary path → verify: WAV production E2E opens through the existing transport.
2. Add browser-native codec fallback → verify: fallback uses a blob URL, connects to the same analysis node, and never performs a network request.
3. Handle autoplay policy without discarding the file → verify: `NotAllowedError` leaves transport visible and instructs the user to tap Play.
4. Bound stalled metadata loading → verify: fallback rejects after 15 seconds and revokes its object URL.

### Phase 3 — close the release-control gap

1. Add deterministic build inspection → verify: `npm run build` prints the verified AudioWorklet chunk path.
2. Test production and development servers independently → verify: CI runs both Playwright configurations.
3. Check shipped dependency exposure → verify: `npm run audit:production` reports zero production vulnerabilities.
4. Run the local manual gate before commit → verify: unit tests, build, production audit, and production preview E2E all pass.

### Phase 4 — real-device acceptance

1. Deploy the patched build → verify: Vercel deployment is tied to the expected commit.
2. Test Android Chromium-family browser with a WAV and representative MP3 → verify: audio is audible, transport advances, and geometry responds.
3. Test microphone permission states → verify: allow starts listening; deny shows permission-specific copy; OS-blocked access shows a device/OS-specific message.
4. Test background/return and Stop → verify: Stop ends microphone tracks and playback; returning to the page does not resurrect a stopped source.
5. Test iOS Safari separately → verify: file playback and microphone meet the same acceptance checks or are documented as unsupported with exact evidence.

## Local-first PowerShell validation

Run from the repository root in this order:

```powershell
npm ci
npm test
npm run build
npm run audit:production
npx playwright install chromium
npm run test:e2e
npx playwright test -c playwright.prod.config.ts
```

Inspect the production bundle directly:

```powershell
Get-ChildItem .\dist\assets\preprocessor.worklet-*.js
Select-String -Path .\dist\assets\*.js -Pattern 'data:video/mp2t'
```

The first command must return exactly one worklet file. The second must return no matches.

After deployment:

```powershell
curl.exe -sSI "https://<production-domain>/" |
    Select-String -Pattern "HTTP/|cross-origin-opener-policy|cross-origin-embedder-policy"
```

## Validation completed in the implementation environment

- `npm test`: 13 test files, 67 tests passed.
- `npm run build`: TypeScript passed; Vite production build passed; worklet bundle verifier passed.
- `npm run audit:production`: zero production vulnerabilities.
- Full `npm audit`: five development-tool findings; remediation requires a breaking Vite upgrade.
- Production Playwright tests were authored but not executed successfully here. The environment
  could not download Playwright Chromium. Its system Chromium is centrally managed with a wildcard
  URL blocklist and audio capture disabled, so it cannot be used as a substitute. The tests remain
  a required local/CI gate, not a claimed pass.
